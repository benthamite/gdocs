This manual describes the features and customization options for the Emacs Lisp package =gdocs=.

* Overview
:PROPERTIES:
:CUSTOM_ID: h:overview
:END:

The =gdocs= package provides bidirectional synchronization between
org-mode files and Google Docs.  It lets you open, edit, create, and
push Google Docs entirely from within Emacs, using org-mode as the
native editing format.  The package handles OAuth 2.0 authentication,
format conversion, incremental diffing, and conflict resolution so that
your org files and their Google Docs counterparts stay in sync.

The package addresses three main workflows:

- *Publish org to Google Docs.*  Create a Google Doc from an existing
  org file to share with collaborators.  The two files remain in sync:
  edits to either propagate to the other.

- *Read and edit shared Google Docs.*  Open a Google Doc shared by
  someone else as an org-mode buffer in Emacs.  Edit without leaving
  Emacs.

- *Import existing Google Docs.*  Import old Google Docs into org
  files.  The files remain synchronized, but you may later choose to
  unlink or delete the Google Doc.

** Architecture
:PROPERTIES:
:CUSTOM_ID: h:architecture
:END:

The package is composed of several modules, each responsible for a
distinct layer of functionality:

- =gdocs.el= :: Entry point, user commands, and the ~gdocs-mode~ minor
  mode ([[#h:minor-mode][Minor mode]]).
- =gdocs-auth.el= :: OAuth 2.0 flow, token management, and
  multi-account support ([[#h:authentication][Authentication]]).
- =gdocs-api.el= :: Asynchronous Google Docs and Google Drive API
  client built on =plz.el= ([[#h:api-client][API client]]).
- =gdocs-convert.el= :: Bidirectional conversion between org-mode and
  Google Docs JSON via an intermediate representation (IR)
  ([[#h:format-conversion][Format conversion]]).
- =gdocs-diff.el= :: Incremental diff engine that computes the minimal
  set of API requests to synchronize changes ([[#h:diff-engine][Diff engine]]).
- =gdocs-sync.el= :: Push/pull orchestration, shadow copy management,
  and conflict detection ([[#h:sync-engine][Sync engine]]).
- =gdocs-merge.el= :: Side-by-side conflict resolution interface
  ([[#h:conflict-resolution][Conflict resolution]]).

All API operations are fully asynchronous via =plz.el= so that Emacs
never blocks during network requests.

** Dependencies
:PROPERTIES:
:CUSTOM_ID: h:dependencies
:END:

The package requires Emacs 29.1 or later and depends on the following
external packages:

- =plz= (0.7+) for asynchronous HTTP requests.
- =dash= (2.19+) for list manipulation utilities.
- =s= (1.13+) for string manipulation utilities.
- =org= (9.6+), which is built into Emacs.

** Quick start
:PROPERTIES:
:CUSTOM_ID: h:quick-start
:END:

1. Obtain OAuth credentials from Google Cloud Console
   ([[#h:obtaining-credentials][Obtaining credentials]]).

2. Configure your credentials in Emacs:

   #+begin_src emacs-lisp
   (setopt gdocs-accounts
           '(("personal" . ((client-id . "YOUR-CLIENT-ID")
                            (client-secret . "YOUR-CLIENT-SECRET")))))
   #+end_src

3. Authenticate: =M-x gdocs-authenticate=.

4. Open an existing Google Doc: =M-x gdocs-open=.

5. Or create a new Google Doc from an org file: =M-x gdocs-create=.

Once a buffer is linked, the ~gdocs-mode~ minor mode activates
automatically.  Changes are pushed to Google Docs on every save (when
~gdocs-auto-push-on-save~ is non-nil), and you can pull remote changes
at any time with =M-x gdocs-pull=.

* User options
:PROPERTIES:
:CUSTOM_ID: h:user-options
:END:

** Account configuration
:PROPERTIES:
:CUSTOM_ID: h:account-configuration
:END:

#+vindex: gdocs-accounts
The user option ~gdocs-accounts~ is an alist of named Google accounts
with their OAuth credentials.  Each entry takes the form
~(NAME . ((client-id . "ID") (client-secret . "SECRET")))~ where NAME
is a string identifying the account (e.g., ="personal"= or ="work"=),
and ~client-id~ and ~client-secret~ are OAuth 2.0 credentials that you
obtain from the Google Cloud Console ([[#h:obtaining-credentials][Obtaining credentials]]).

When ~gdocs-accounts~ is ~nil~ or has only one entry, commands that
require an account use it automatically without prompting.  When
multiple accounts are configured, commands prompt with
~completing-read~ to select one.  You would configure multiple accounts
when you need to access Google Docs from different Google Workspace
organizations (for example, a personal account and a work account with
separate OAuth credentials).

The default value is ~nil~.  You must configure at least one account
before using any =gdocs= commands.


*** Obtaining credentials
:PROPERTIES:
:CUSTOM_ID: h:obtaining-credentials
:END:

To use =gdocs=, you need an OAuth 2.0 /client ID/ and /client secret/
from the Google Cloud Console.  These credentials identify your
application to Google's servers and allow it to request access to your
Google Docs and Drive data on your behalf.  They are not passwords ---
they do not grant access by themselves.  The actual authorization
happens when you run ~gdocs-authenticate~ and grant consent in your
browser.

Follow these steps to create your credentials:

1. Go to the [[https://console.cloud.google.com/][Google Cloud Console]] and sign in with the Google account
   you want to use with =gdocs=.

2. Create a new project (or select an existing one):
   - Click the project dropdown at the top of the page.
   - Click *New Project*.
   - Enter a name (e.g., "Emacs gdocs") and click *Create*.

3. Enable the required APIs.  Go to *APIs & Services > Library* and
   enable both of these:
   - *Google Docs API* (search for "Google Docs API").
   - *Google Drive API* (search for "Google Drive API").

4. Configure the OAuth consent screen.  Go to *APIs & Services > OAuth
   consent screen*:
   - Select *External* as the user type (unless you have a Google
     Workspace organization and want to restrict access to internal
     users).
   - Fill in the required fields: app name (e.g., "Emacs gdocs") and
     your email address for the support and developer contact fields.
   - On the *Scopes* page, add these two scopes:
     - =https://www.googleapis.com/auth/documents=
     - =https://www.googleapis.com/auth/drive=
   - On the *Test users* page, add your own email address.  While the
     app is in "Testing" status (the default), only listed test users
     can authorize it.
   - Click *Save and Continue* through the remaining pages.

5. Create the OAuth credentials.  Go to *APIs & Services >
   Credentials*:
   - Click *Create Credentials > OAuth client ID*.
   - Select *Desktop app* as the application type.
   - Enter a name (e.g., "Emacs gdocs") and click *Create*.
   - Google will display your *Client ID* and *Client secret*.  Copy
     both values.

6. Configure =gdocs= with the credentials you just obtained:

   #+begin_src emacs-lisp
   (setopt gdocs-accounts
           '(("personal" . ((client-id . "123456789-abcdef.apps.googleusercontent.com")
                            (client-secret . "GOCSPX-abcdefghijklmnop")))))
   #+end_src

   Replace the example values with your actual client ID and client
   secret.  The name ="personal"= is arbitrary; use whatever label
   makes sense to you.

7. Run =M-x gdocs-authenticate=.  Your browser will open to a Google
   consent screen.  Grant access, and =gdocs= will store the resulting
   tokens securely in ~gdocs-token-directory~
   ([[#h:account-configuration][Account configuration]]).

Note: while the app remains in "Testing" status, Google will show a
warning screen during authorization ("Google hasn't verified this
app").  Click *Continue* to proceed.  You do not need to publish the
app for personal use --- "Testing" status works indefinitely for the
test users you listed in step 4.

#+vindex: gdocs-token-directory
The user option ~gdocs-token-directory~ specifies the directory where
OAuth token files are stored.  Each account's tokens are saved as a
JSON file named =ACCOUNT-NAME.json= with restrictive file permissions
(600) to protect the sensitive token data.  Parent directories are
created automatically when needed.

The default value is the =gdocs/tokens/= subdirectory of
~user-emacs-directory~.  You would change this if you want to store
tokens in a location managed by a secrets tool or a different
directory structure.

** Sync behavior
:PROPERTIES:
:CUSTOM_ID: h:sync-behavior
:END:

#+vindex: gdocs-auto-push-on-save
The user option ~gdocs-auto-push-on-save~ controls whether changes are
automatically pushed to Google Docs whenever you save a linked org
buffer.  When non-nil, saving the buffer triggers an asynchronous push
of any changes detected since the last sync.

The default value is ~t~.  Set this to ~nil~ if you prefer to push
changes manually with ~gdocs-push~ ([[#h:pushing-and-pulling][Pushing and pulling]]).  Automatic
pushing is suppressed when the buffer is in a conflict or error state,
regardless of this setting.

#+vindex: gdocs-auto-pull-on-open
The user option ~gdocs-auto-pull-on-open~ controls whether a linked org
buffer automatically pulls the latest remote content when opened.  When
non-nil, opening a file that has ~gdocs-mode~ active triggers an
immediate pull to ensure you start editing the most recent version.

The default value is ~nil~.  Enable this if you frequently collaborate
on documents and want to always start with the latest remote state.
When disabled, the buffer shows whatever was last saved locally, and you
must run ~gdocs-pull~ manually to fetch updates.

** File storage
:PROPERTIES:
:CUSTOM_ID: h:file-storage
:END:

#+vindex: gdocs-directory
The user option ~gdocs-directory~ specifies the default directory where
newly imported org files are saved.  When you use ~gdocs-open~ to fetch
a Google Doc for the first time, the resulting org file is written to
this directory with a filename derived from the document title.

The default value is =~/org/gdocs/=.  The directory is created
automatically if it does not exist.  If you already have an org file
and use ~gdocs-create~ or ~gdocs-link~, the file stays at its current
location regardless of this setting.

* Commands
:PROPERTIES:
:CUSTOM_ID: h:commands
:END:

** Opening and creating documents
:PROPERTIES:
:CUSTOM_ID: h:opening-and-creating
:END:

#+findex: gdocs-open
#+findex: gdocs-create
The command ~gdocs-open~ fetches a Google Doc and opens it as an
org-mode buffer.  It prompts for a document ID or a full Google Docs
URL (e.g., =https://docs.google.com/document/d/DOCUMENT-ID/edit=).  If
multiple accounts are configured in ~gdocs-accounts~
([[#h:account-configuration][Account configuration]]), it prompts you to select one.  The document is
converted from Google Docs JSON to org-mode format via the intermediate
representation ([[#h:format-conversion][Format conversion]]), saved to ~gdocs-directory~
([[#h:file-storage][File storage]]) with a sanitized filename, and displayed in a buffer
with ~gdocs-mode~ active.

The command ~gdocs-create~ does the reverse: it creates a new Google
Doc from the current org buffer's content.  The title defaults to the
buffer's =#+TITLE= keyword, falling back to the buffer name.  After
creating the document, the buffer is linked to it and ~gdocs-mode~ is
activated.  Subsequent saves push changes to the newly created Doc.

Both commands are autoloaded, so they are available without explicitly
loading the package.

** Pushing and pulling
:PROPERTIES:
:CUSTOM_ID: h:pushing-and-pulling
:END:

#+findex: gdocs-push
#+findex: gdocs-pull
The command ~gdocs-push~ pushes local changes to the linked Google Doc.
If a shadow copy exists (i.e., the buffer has been synced at least once
in the current session), the push is incremental: only the differences
between the current buffer and the shadow are sent as Google Docs API
~batchUpdate~ requests ([[#h:diff-engine][Diff engine]]).  If no shadow exists (for
example, after restarting Emacs), the push performs a full document
replacement.

When ~gdocs-auto-push-on-save~ is non-nil, ~gdocs-push~ is called
automatically after every save.  Pushes are serialized: if a push is
already in progress when another is triggered, the second push is
queued and executed after the first completes.

The command ~gdocs-pull~ fetches the latest version of the linked
Google Doc.  It first checks the document's revision ID via the Google
Drive API; if the revision matches the locally stored one, it reports
"Already up to date."  If the revision differs, it fetches the full
document, converts it to org-mode format, and either replaces the
buffer content silently (when there are no local modifications) or
triggers the conflict resolution interface ([[#h:conflict-resolution][Conflict resolution]]).

** Linking and unlinking
:PROPERTIES:
:CUSTOM_ID: h:linking-and-unlinking
:END:

#+findex: gdocs-link
#+findex: gdocs-unlink
The command ~gdocs-link~ links the current org buffer to an existing
Google Doc.  It prompts for a document ID or URL and, if needed, an
account.  Linking writes file-local variables (=gdocs-document-id= and
=gdocs-account=) to the end of the org file and triggers an initial
pull to establish the sync baseline.

The command ~gdocs-unlink~ removes the file-local variables and clears
all buffer-local sync state, turning the org file back into a
standalone document.  The file content is preserved; only the sync
metadata is removed.

** Status and browser
:PROPERTIES:
:CUSTOM_ID: h:status-and-browser
:END:

#+findex: gdocs-status
#+findex: gdocs-open-in-browser
The command ~gdocs-status~ displays the current sync status in the echo
area.  It shows the linked document ID, account name, sync status
(synced, modified, pushing, conflict, or error), revision ID, and the
timestamp of the last sync.  This is useful for quickly verifying
whether the buffer is in sync without inspecting the modeline.

The command ~gdocs-open-in-browser~ opens the linked Google Doc in your
default web browser using ~browse-url~.  It constructs the URL from
the stored document ID.  This is convenient when you need to check
formatting, share the link with someone, or use Google Docs features
that are not yet supported in =gdocs=.

** Authentication
:PROPERTIES:
:CUSTOM_ID: h:authentication
:END:

#+findex: gdocs-authenticate
#+findex: gdocs-logout
The command ~gdocs-authenticate~ initiates the OAuth 2.0 flow for a
Google account ([[#h:oauth-flow][OAuth flow]]).  If multiple accounts are configured, it
prompts you to select one.  The command opens your browser to Google's
consent screen, and a temporary local HTTP server receives the
authorization callback.  Upon successful authentication, tokens are
stored in ~gdocs-token-directory~ ([[#h:account-configuration][Account configuration]]).

The command ~gdocs-logout~ deletes the stored token file for an
account, effectively logging you out.  You will need to run
~gdocs-authenticate~ again to re-authorize.

Both commands are autoloaded.

** Conflict resolution
:PROPERTIES:
:CUSTOM_ID: h:conflict-resolution-commands
:END:

#+findex: gdocs-merge-next-hunk
#+findex: gdocs-merge-previous-hunk
#+findex: gdocs-merge-accept-local
#+findex: gdocs-merge-accept-remote
#+findex: gdocs-merge-edit-hunk
#+findex: gdocs-merge-accept-all-local
#+findex: gdocs-merge-accept-all-remote
#+findex: gdocs-merge-finalize
#+findex: gdocs-merge-abort
When a pull detects that both local and remote changes exist, the
conflict resolution interface opens in a dedicated =*gdocs-merge*=
buffer using ~gdocs-merge-mode~ ([[#h:merge-mode][Merge mode]]).  The following commands
are available in the merge buffer:

- ~gdocs-merge-next-hunk~ (=n=) moves to the next unresolved conflict
  hunk.
- ~gdocs-merge-previous-hunk~ (=p=) moves to the previous unresolved
  hunk.
- ~gdocs-merge-accept-local~ (=a=) accepts the local version of the
  current hunk.
- ~gdocs-merge-accept-remote~ (=b=) accepts the remote version of the
  current hunk.
- ~gdocs-merge-edit-hunk~ (=e=) opens a temporary org-mode buffer
  pre-filled with the local text so you can edit the merged result
  manually.  Press =C-c C-c= in the edit buffer to accept.
- ~gdocs-merge-accept-all-local~ (=A=) resolves all unresolved hunks
  with the local version.
- ~gdocs-merge-accept-all-remote~ (=B=) resolves all unresolved hunks
  with the remote version.
- ~gdocs-merge-finalize~ (=C-c C-c=) finalizes the merge once all
  hunks are resolved, applies the merged content to the original
  buffer, and triggers a push.  It signals an error if any hunks remain
  unresolved.
- ~gdocs-merge-abort~ (=C-c C-k= or =q=) aborts the merge after
  confirmation, leaving the buffer unchanged.

* Functions
:PROPERTIES:
:CUSTOM_ID: h:functions
:END:

** Authentication
:PROPERTIES:
:CUSTOM_ID: h:auth-functions
:END:

#+findex: gdocs-auth-select-account
#+findex: gdocs-auth-get-access-token
The function ~gdocs-auth-select-account~ selects a Google account from
~gdocs-accounts~.  If only one account is configured, it returns that
account's name without prompting.  If multiple accounts exist, it
prompts via ~completing-read~ with an optional PROMPT argument.  It
signals an error if ~gdocs-accounts~ is ~nil~.  This function is used
internally by most commands but is also available for use in custom
workflows or hooks.

The function ~gdocs-auth-get-access-token~ ensures a valid OAuth access
token for a given account, then calls a callback with the token string.
If the token is expired or about to expire (within a 60-second safety
margin), it refreshes the token asynchronously before calling the
callback.  If no token exists, it initiates the full OAuth flow.  This
function is the primary entry point for all API operations that need
authentication.

** API client
:PROPERTIES:
:CUSTOM_ID: h:api-functions
:END:

*** Google Docs API
:PROPERTIES:
:CUSTOM_ID: h:docs-api-functions
:END:

#+findex: gdocs-api-get-document
#+findex: gdocs-api-batch-update
#+findex: gdocs-api-create-document
The function ~gdocs-api-get-document~ fetches a Google Docs document by
its ID.  It calls CALLBACK with the parsed JSON document structure.
This is used internally by ~gdocs-pull~ to retrieve the remote document
state.

The function ~gdocs-api-batch-update~ sends a list of ~batchUpdate~
requests to a document, which is how all modifications (insertions,
deletions, style changes) are applied to Google Docs.  The REQUESTS
argument is a list of request alists as specified by the Google Docs
API.  This function is called by both the incremental diff engine and
full replacement paths.

The function ~gdocs-api-create-document~ creates a new empty Google
Docs document with a given title.  The callback receives the response
JSON, which includes the new =documentId=.

All three functions accept an optional ACCOUNT argument; when ~nil~,
the default account is selected automatically
([[#h:account-configuration][Account configuration]]).

*** Google Drive API
:PROPERTIES:
:CUSTOM_ID: h:drive-api-functions
:END:

#+findex: gdocs-api-get-file-metadata
#+findex: gdocs-api-list-files
#+findex: gdocs-api-upload-image
#+findex: gdocs-api-search-files
The function ~gdocs-api-get-file-metadata~ fetches metadata for a
Google Drive file, including its =id=, =name=, =modifiedTime=,
=version=, and =headRevisionId=.  The revision ID is used by the pull
mechanism to detect whether the remote document has changed
([[#h:pushing-and-pulling][Pushing and pulling]]).

The function ~gdocs-api-list-files~ lists Google Drive files matching a
query string (using Drive's query syntax).  It supports pagination via
an optional PAGE-TOKEN argument.

The function ~gdocs-api-upload-image~ uploads a local image file to
Google Drive using a multipart upload.  It determines the MIME type
from the file extension (PNG, JPEG, GIF, WebP, and SVG are supported).
An optional FOLDER-ID argument specifies the parent folder in Drive.

The function ~gdocs-api-search-files~ performs a full-text search across
Google Drive by wrapping ~gdocs-api-list-files~ with a
=fullText contains= query.

** Format conversion
:PROPERTIES:
:CUSTOM_ID: h:conversion-functions
:END:

#+findex: gdocs-convert-org-buffer-to-ir
#+findex: gdocs-convert-org-string-to-ir
#+findex: gdocs-convert-org-buffer-to-segments
#+findex: gdocs-convert-ir-to-org
#+findex: gdocs-convert-docs-json-to-ir
#+findex: gdocs-convert-ir-to-docs-requests
All conversion passes through a flat list of intermediate
representation (IR) element plists, which serves as the canonical
interchange format shared with the diff engine ([[#h:diff-engine][Diff engine]]).

The function ~gdocs-convert-org-buffer-to-ir~ parses the current org
buffer using ~org-element-parse-buffer~ and returns an IR element list.
The related function ~gdocs-convert-org-string-to-ir~ does the same but
accepts an org-mode string rather than operating on the current buffer.

The function ~gdocs-convert-org-buffer-to-segments~ is a variant of
~gdocs-convert-org-buffer-to-ir~ designed for the three-way merge.
Instead of returning a flat IR list, it returns a plist with four keys:
~:ir~ (the full IR element list), ~:segments~ (a list of segment plists,
each pairing an ~:ir-element~ with its ~:org-text~ region), ~:preamble~
(buffer text before the first body element, such as the =#+TITLE:=
line), and ~:postamble~ (text after the last element, typically the
file-local variables block).  By preserving the mapping between IR
elements and their original org text, the three-way merge can retain
org-only metadata (property drawers, TODO keywords, tags) that would
otherwise be lost during conversion ([[#h:sync-engine][Sync engine]]).

The function ~gdocs-convert-ir-to-org~ converts an IR element list back
to a well-formatted org-mode string.  This is the inverse of
~gdocs-convert-org-buffer-to-ir~ and is used when pulling remote
changes.

The function ~gdocs-convert-docs-json-to-ir~ converts a Google Docs API
JSON response (an alist with keys like =body=, =title=, =lists=, and
=namedRanges=) to an IR element list.  It walks the document structure
and maps each structural element to its IR equivalent.

The function ~gdocs-convert-ir-to-docs-requests~ converts an IR element
list to a list of Google Docs ~batchUpdate~ request alists.  It tracks
a running UTF-16 insertion index starting at 1 (the beginning of the
Google Docs body).  This function is used for full document replacement
and by the diff engine for incremental insertions.

** Diff engine
:PROPERTIES:
:CUSTOM_ID: h:diff-functions
:END:

#+findex: gdocs-diff-generate
The function ~gdocs-diff-generate~ is the main entry point of the diff
engine.  It compares two IR element lists (OLD-IR and NEW-IR) and
returns a list of ~batchUpdate~ request plists representing the minimal
changes needed.  Internally, it computes content-based keys for each
element, runs a longest-common-subsequence (LCS) algorithm to identify
kept, deleted, inserted, and modified elements, and generates the
appropriate API requests.  When more than 50% of elements have changed,
it falls back to a full document replacement to avoid overly complex
request sequences.

An optional START-INDEX argument specifies the UTF-16 code unit offset
where the first IR element begins in the document body (default 1).
The sync engine uses this to account for title elements that precede the
diffable body content ([[#h:sync-engine][Sync engine]]).  Requests are ordered for correct
index stability: deletions and modifications processed from highest
index to lowest, with insertions interleaved so that earlier operations
do not shift indices of later ones.

** Conflict resolution
:PROPERTIES:
:CUSTOM_ID: h:merge-functions
:END:

#+findex: gdocs-merge-start
The function ~gdocs-merge-start~ opens the conflict resolution
interface.  It takes two org-mode strings (local and remote) and a
callback function.  It computes hunks by running an LCS diff on the
line-level differences, sets up the =*gdocs-merge*= buffer with
color-coded overlays for conflicts, and displays it.  When the user
finalizes the merge, the callback is called with the resulting merged
org string.

This function is called internally by the sync engine when a pull
detects both local and remote changes ([[#h:conflict-resolution][Conflict resolution]]).

* Authentication
:PROPERTIES:
:CUSTOM_ID: h:authentication-details
:END:

** OAuth flow
:PROPERTIES:
:CUSTOM_ID: h:oauth-flow
:END:

The OAuth 2.0 flow works as follows:

1. The user runs =M-x gdocs-authenticate=.
2. A temporary local HTTP server starts on a random high port to
   receive the authorization callback.
3. The user's default browser opens to Google's OAuth consent screen.
4. After the user grants consent, Google redirects to the local server
   with an authorization code.
5. The package exchanges the code for access and refresh tokens.
6. Tokens are stored in ~gdocs-token-directory~ with 600 permissions.

The package requests two OAuth scopes:

- =https://www.googleapis.com/auth/documents= for reading and writing
  Google Docs.
- =https://www.googleapis.com/auth/drive= for listing, browsing, and
  creating files in Google Drive.

** Token management
:PROPERTIES:
:CUSTOM_ID: h:token-management
:END:

Access tokens are refreshed automatically before each API call when
they are expired or within 60 seconds of expiry.  The refresh uses the
stored refresh token and happens asynchronously, so Emacs is never
blocked.  Each token file also stores the =client_id= and
=client_secret= so that token refreshes work without re-reading
~gdocs-accounts~.

If no token file exists for an account (e.g., after running
~gdocs-logout~), the next API operation triggers a fresh OAuth flow.

** Multi-account support
:PROPERTIES:
:CUSTOM_ID: h:multi-account
:END:

Each entry in ~gdocs-accounts~ is an independent account with its own
OAuth credentials and token file.  Linked org files remember which
account they are associated with via the =gdocs-account= file-local
variable.  When you open a linked file, the stored account is used
automatically.

You would configure custom OAuth credentials per account if you want to
avoid shared rate limits or if your organization requires its own OAuth
application.

** Error handling and retries
:PROPERTIES:
:CUSTOM_ID: h:error-handling
:END:

All API requests include automatic retry logic for transient errors.
When the server returns HTTP 429 (rate limited), 500 (internal server
error), or 503 (service unavailable), the request is retried with
exponential backoff up to three times.  If the response includes a
=Retry-After= header, its value is used as the delay; otherwise, the
delay follows the pattern 1s, 2s, 4s, capped at 30 seconds.

Authentication failures (HTTP 401 or 403) produce a clear error message
directing you to run ~gdocs-authenticate~ to re-authorize.

** Progress indication
:PROPERTIES:
:CUSTOM_ID: h:progress-indication
:END:

While API requests are in flight, the global modeline displays
=[GDocs: syncing...]= to indicate that a network operation is in
progress.  The indicator disappears when all pending requests complete.

* Format conversion
:PROPERTIES:
:CUSTOM_ID: h:format-conversion
:END:

** Supported element mapping
:PROPERTIES:
:CUSTOM_ID: h:element-mapping
:END:

The converter handles the following bidirectional mapping between
org-mode elements and Google Docs constructs:

| Org element                    | Google Docs equivalent               |
|--------------------------------+--------------------------------------|
| =#+TITLE:=                     | Title paragraph style                |
| =* Heading= (level 1--6)      | Heading 1--6 paragraph style         |
| =* Heading= (level 7+)        | Heading 6 (flattened)                |
| Body text / paragraphs         | Normal text paragraphs               |
| =*bold*=                       | Bold text run                        |
| =/italic/=                     | Italic text run                      |
| =_underline_=                  | Underline text run                   |
| =+strikethrough+=              | Strikethrough text run               |
| ~=verbatim=~ / =~code~=       | Monospace font text run              |
| =[[url][description]]=         | Hyperlink                            |
| Unordered lists (=-=, =+=)    | Bullet list                          |
| Ordered lists (=1.=, =1)=)    | Numbered list                        |
| Nested lists                   | Nested list with nesting level       |
| Checkbox items (=[ ]=, =[X]=) | Checklist items                      |
| Tables                         | Google Docs table                    |
| Horizontal rules (=-----=)    | Horizontal rule (section break)      |
| Block quotes                   | Indented paragraph or custom style   |
| Footnotes (=[fn:1]=)          | Google Docs footnote                 |
| Images (=[[file:img.png]]=)   | Inline image (uploaded to Drive)     |

#+vindex: gdocs-convert-monospace-font-families
Google Docs has no native "code" text style, so the converter uses a
font-based heuristic: when pulling, any text run whose font family
appears in the variable ~gdocs-convert-monospace-font-families~ is
treated as code (mapped to =~code~= in org-mode).  When pushing, code
runs are set to the first font in the list (=Courier New= by default).
The recognized families are =Courier New=, =Consolas=, =Source Code
Pro=, =Roboto Mono=, and =Courier=.  You can customize this list if
your documents use a different monospace font.

** Intermediate representation
:PROPERTIES:
:CUSTOM_ID: h:intermediate-representation
:END:

All conversion passes through a flat list of IR element plists rather
than directly translating between org-mode and Google Docs JSON.  Each
IR element has a =:type= (e.g., ~paragraph~, ~table~,
~horizontal-rule~), a =:style= (e.g., ~normal~, ~heading-1~), and
=:contents= (a list of text runs with formatting properties like
=:bold=, =:italic=, =:link=, etc.).

This architecture means the diff engine ([[#h:diff-engine][Diff engine]]) only needs to
understand the IR format, not both org-mode and Google Docs JSON
independently.

** Org-only constructs
:PROPERTIES:
:CUSTOM_ID: h:org-only-constructs
:END:

Certain org-mode constructs have no direct Google Docs equivalent.
These are preserved as special markers in the Google Doc so they survive
round-trips:

- *TODO keywords*: rendered as styled text with a preservation comment.
- *Tags*: same treatment as TODO keywords.
- *Timestamps* (=SCHEDULED:=, =DEADLINE:=, active/inactive): preserved
  as markers.
- *Properties and drawers*: preserved as markers, not rendered visually.
- *Source code blocks*: rendered as monospace-formatted text with the
  full org syntax in a comment.
- *LaTeX fragments*: rendered as monospace text with the raw LaTeX in a
  comment.
- *Org keywords* (=#+AUTHOR:=, =#+DATE:=, etc.): preserved as markers
  at the top of the document.
- *Custom IDs and anchors*: preserved as comments.

** Google Docs-only constructs
:PROPERTIES:
:CUSTOM_ID: h:docs-only-constructs
:END:

Similarly, some Google Docs constructs have no org equivalent and are
preserved as opaque org markers:

- Page breaks: =#+GDOCS_PAGE_BREAK:=
- Headers/footers: =#+GDOCS_HEADER:= / =#+GDOCS_FOOTER:=
- Page numbers: =#+GDOCS_PAGE_NUMBER:=
- Table of contents: =#+GDOCS_TOC:=
- Embedded drawings: =#+GDOCS_DRAWING:=
- Charts: =#+GDOCS_CHART:=
- Named styles / custom formatting: preserved as inline markers

* Sync engine
:PROPERTIES:
:CUSTOM_ID: h:sync-engine
:END:

** Metadata storage
:PROPERTIES:
:CUSTOM_ID: h:metadata-storage
:END:

Each linked org file stores sync metadata in file-local variables at
the end of the file:

#+begin_example
# Local Variables:
# gdocs-document-id: "1BxiMVs0XRA5nFMdKvBdBZjgmUUqptlbs74OgVE2upms"
# gdocs-account: "personal"
# gdocs-revision-id: "ALm37BVT..."
# gdocs-last-sync: "2024-01-15T10:30:00Z"
# End:
#+end_example

These variables are declared safe for file-local use (via
~safe-local-variable~) so Emacs does not prompt for confirmation when
opening a linked file.

** Shadow copy
:PROPERTIES:
:CUSTOM_ID: h:shadow-copy
:END:

The sync engine maintains a buffer-local /shadow IR/: the intermediate
representation of the last-synced state.  This shadow enables
incremental diffs on push and conflict detection on pull.

On the first open of a linked file after an Emacs restart, the shadow
is ~nil~.  In this case, the next push performs a full document
replacement rather than an incremental diff, and a message warns: "No
sync baseline in memory."  Alternatively, if ~gdocs-auto-pull-on-open~
is non-nil, the pull establishes the shadow baseline automatically.

** Push serialization
:PROPERTIES:
:CUSTOM_ID: h:push-serialization
:END:

Pushes are serialized to prevent race conditions.  If a push is already
in flight when another is triggered (e.g., by a rapid save), the second
push is queued and executes automatically when the first completes.
This ensures that the shadow copy is always consistent with what was
last sent to the API.

* Diff engine
:PROPERTIES:
:CUSTOM_ID: h:diff-engine
:END:

The incremental diff engine computes the minimal set of Google Docs API
~batchUpdate~ requests to transform the remote document from its
last-known state to the current org state.

** Approach
:PROPERTIES:
:CUSTOM_ID: h:diff-approach
:END:

1. Produce content-based keys for each element in the old and new IRs.
   Keys capture type, style, text content, and formatting, so elements
   with identical content produce identical keys.

2. Run a longest-common-subsequence (LCS) algorithm on the key lists to
   identify which elements are kept, deleted, inserted, or modified.
   Adjacent delete+insert pairs of the same type are collapsed into
   modify operations.

3. Map operations to API requests:
   - *Insert*: ~InsertText~, ~InsertTable~, ~CreateParagraphBullets~, etc.
   - *Delete*: ~DeleteContentRange~.
   - *Modify*: for style-only changes, ~UpdateParagraphStyle~; for
     formatting-only changes, ~UpdateTextStyle~; for content changes,
     delete + re-insert.

4. Order requests correctly: deletions in reverse index order (highest
   first), then insertions in forward order.  This is necessary because
   the Google Docs API processes requests sequentially and earlier
   operations shift later indices.

** Fallback
:PROPERTIES:
:CUSTOM_ID: h:diff-fallback
:END:

When more than 50% of elements have changed and the document has more
than two elements, the diff engine falls back to a full document
replacement (delete all + insert all) to avoid producing an overly
complex or fragile request sequence.

* Conflict resolution
:PROPERTIES:
:CUSTOM_ID: h:conflict-resolution
:END:

When a pull detects that the buffer has local modifications and the
remote revision has changed, the conflict resolution UI opens
automatically.  The =*gdocs-merge*= buffer displays all hunks
(computed via a line-level LCS diff), with conflicts shown in a
=<<<< LOCAL= / =====/ =>>>> REMOTE= format.

Common (matching) lines are displayed without highlighting.  Conflict
hunks use color-coded overlays: ~gdocs-merge-remote-face~ for
unresolved conflicts, ~gdocs-merge-resolved-face~ for resolved ones.

After resolving all hunks and finalizing with =C-c C-c=, the merged
content replaces the buffer, the shadow IR is updated, and the buffer
is saved (which triggers a push if auto-push is enabled).

* Minor mode
:PROPERTIES:
:CUSTOM_ID: h:minor-mode
:END:

#+findex: gdocs-mode
The minor mode ~gdocs-mode~ is activated on org buffers linked to a
Google Doc.  It provides:

- Automatic push on save (via ~after-save-hook~) when
  ~gdocs-auto-push-on-save~ is non-nil.
- Modeline status display showing the sync state:
  - =GDocs:synced= when in sync.
  - =GDocs:modified= when local changes have not been pushed.
  - =GDocs:pushing= while a push is in progress.
  - =GDocs:conflict= when remote changes conflict with local ones.
  - =GDocs:error= when the last sync failed.
- Keybindings under the =C-c g= prefix ([[#h:keybindings][Keybindings]]).

The mode activates automatically when you open an org file that has the
=gdocs-document-id= file-local variable set (via a hook on
~org-mode-hook~).  You can also toggle it manually.

** Keybindings
:PROPERTIES:
:CUSTOM_ID: h:keybindings
:END:

The ~gdocs-mode-map~ provides the following keybindings:

| Key       | Command                | Description                        |
|-----------+------------------------+------------------------------------|
| =C-c g p= | ~gdocs-push~           | Push local changes                 |
| =C-c g l= | ~gdocs-pull~           | Pull remote changes                |

| =C-c g o= | ~gdocs-open-in-browser~ | Open document in browser           |
| =C-c g u= | ~gdocs-unlink~         | Unlink from Google Doc             |

* Merge mode
:PROPERTIES:
:CUSTOM_ID: h:merge-mode
:END:

~gdocs-merge-mode~ is a major mode derived from ~special-mode~ used in
the =*gdocs-merge*= buffer during conflict resolution.  Its keybindings
are documented in [[#h:conflict-resolution-commands][Conflict resolution]].

The mode defines three faces for visual differentiation:

- ~gdocs-merge-local-face~ (green background) for local changes.
- ~gdocs-merge-remote-face~ (red background) for remote changes.
- ~gdocs-merge-resolved-face~ (blue background) for resolved hunks.
